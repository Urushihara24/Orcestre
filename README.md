# Multi Friends Epic Bot

Telegram-бот для управления пулом аккаунтов Epic Games:
- импорт аккаунтов-отправителей;
- авторизация через Epic OAuth и сохранение `device_auth`;
- работа с несколькими целями;
- отправка заявок в друзья на ники получателя с растяжением по времени;
- проверка статусов, отзыв заявок, удаление из друзей.

## 3. Что настраивается (без лишнего)

Интерфейс управления: **inline-кнопки** (кнопки в сообщениях бота).
Reply-клавиатура отключена.

### 3.1. Навигация (inline)

- `cmd_reply_nav` — единая точка входа для текстов/inline-экшенов.
- `act:*` callback-кнопки конвертируются в текстовый action и проходят через тот же роутинг.
- Маршрутизация разбита на helper-блоки:
  - `_handle_auth_operator_nav` — ограниченный режим auth-оператора;
  - `_dispatch_admin_nav` — основной admin-dispatch;
  - `_handle_targets_goals_actions` / `_handle_goal_edit_shortcuts` / `_handle_settings_and_manage_actions` / `_handle_accounts_actions`.
- Для неизвестного текста есть fallback в текущее меню-контекст (`_show_menu_by_current_context`), чтобы пользователь не “зависал”.

### В цели
- `🔄 Лимит` — лимит отправок на 1 аккаунт-отправитель в сутки (в рамках цели).
- `🎯 На ник` — сколько отправителей нужно на один ник получателя.
- `⏱️ Джиттер` — интервал между заявками одного отправителя.
- `🕐 Окна` — рабочие окна цели (всегда по МСК).
- `🔁 Лимит повторных проверок` — сколько уже отправленных заявок можно перепроверять за сутки (отдельно от отправки).
- `📅 Ежедневный повтор` — включить/выключить повтор цели на следующие дни.
  - при включении лимит `🎯 На ник` применяется по суткам (новый день = новое покрытие отправителями).

### Глобально
- `🛡️ API лимиты` — на каждый аккаунт:
  - минимальный интервал между API-запросами;
  - максимум запросов в час;
  - максимум запросов в сутки.
- `📌 Прокси`
- `👤 Доступ auth` (ограниченные операторы авторизации)
- `🔄 Обновить ники Epic` (в разделе `👥 Аккаунты`) — подтянуть актуальные `displayName` из Epic для списка аккаунтов.

## 4. Логика отправки

В `🎯 Цель -> ✏️ Редактировать цель -> 🔀 Алгоритм отправки` доступны 2 режима:
1. `Сначала отправитель -> все ники` (слойный режим):
   - первый аккаунт идёт по всем никам цели;
   - затем второй, третий и т.д.
2. `Сначала ник -> все отправители`:
   - сначала один ник покрывается всеми нужными отправителями;
   - затем следующий ник.

В `🎯 Цель -> ✏️ Редактировать цель -> 🎲 Порядок отправителей`:
1. `По порядку (ID аккаунтов)` — отправители идут строго по возрастанию ID.
2. `Случайный порядок` — порядок отправителей перемешивается при каждом новом распределении.

Планировщик растягивает задачи по рабочему окну цели, с учётом:
- `⏱️ Джиттер` цели;
- `🛡️ API лимиты` на каждый аккаунт;
- суточного лимита цели (`🔄 Лимит`).

Если `📅 Ежедневный повтор` включен, на следующий день цель продолжает покрытие по дневным правилам (новые send-задачи планируются заново в рамках суточного лимита).

## 5. Почему может быть меньше/больше

### Меньше, чем ожидалось
- не хватает активных авторизованных отправителей;
- часть отправителей вне окна/в лимите/в прогреве;
- API-лимиты ограничили скорость;
- цель ещё не дошла до полного покрытия.

### Больше, чем ожидалось
- раньше уже были созданные задачи до изменения лимита;
- цель была перезапущена с уже существующей историей задач.

При изменении `🎯 На ник` бот:
1. обновляет `required_senders` у ников цели;
2. очищает старые `queued/postponed` send-задачи по цели;
3. строит очередь заново под новый лимит на ники получателя.

Точно так же очередь цели автоматически перестраивается при изменении:
- `🔄 Лимит` (на аккаунт/сутки в цели),
- `⏱️ Джиттер`,
- `🪟 Окна`,
- `🔀 Алгоритм отправки`,
- `🎲 Порядок отправителей`,
- `📅 Ежедневный повтор`.

## 6. Статистика и контроль

- `📊 Статистика цели` — сводка по выбранной цели.
- `📄 Статусы по никам` — по каждому нику получателя:
  - `отправлено`
  - `принято`
  - `в друзьях сейчас` (по последним `check_status`, фактическая текущая дружба)
  - `отклонено`
  - очереди `send/check`
- `👀 Отправители по нику` — кто именно отправлял на выбранный ник получателя.
- `⚡ Форс-цикл с аккаунта` — вручную выполнить один цикл отправки:
  выбранный аккаунт отправляет по всем никам текущей цели один проход.
- `🎲 Форс-цикл (рандом)` — то же самое, но аккаунт выбирается случайно из доступных.
  - используются только активные авторизованные аккаунты;
  - пропускаются пары, где уже есть активная задача или уже был `DONE` для `аккаунт+ник`.

Это основной способ понять “сколько реально отправлено” без ручной проверки аккаунтов.

## 7. Отзыв заявок / удаление из друзей

В меню выбранной цели:
- `↩️ Отозвать заявки`
- `🗑️ Удалить из друзей`
- `⛔ Остановить операцию`

В прогрессе показываются:
- `Успешно`
- `Пропущено`
- `Пропущено (без auth / не подходит статус)` — расшифровка причины пропуска.

## 8. Повторные проверки (recheck)

- Recheck идет отдельным лимитом (`🔁 Лимит повторных проверок`) и не смешивается с лимитом отправок.
- Проверяются ники в статусах `pending/sent/accepted`.
- Если после `accepted` дружба пропала, бот ставит повторную отправку (`send_request`) автоматически.

## 8.1. Окна будни/выходные

Можно задать разные окна в одной цели несколькими строками:
```text
days=1,2,3,4,5 from=12:00 to=20:00
days=6,7 from=10:00 to=18:00
```
Или `24/7`.

## 9. Ограниченный доступ (auth-операторы)

`⚙️ Настройки -> 👤 Доступ auth`:
- `➕ Добавить ID auth`
- `➖ Удалить ID auth`
- `📋 Список ID auth`
- `🗑 Очистить ID auth`

Auth-оператор видит только:
- список аккаунтов;
- авторизацию Epic по ссылке.

## 13. Деплой на сервер (Ubuntu 22.04)

1. Залить проект на сервер (без `venv`, кэшей и локальных артефактов).
2. На сервере в папке проекта выполнить:

```bash

docker compose -f docker-compose.prod.yml ps
docker compose -f docker-compose.prod.yml logs --tail=200 app
```

3. Быстрый post-check:
- бот отвечает на `/start`;
- `🎯 Цели -> 📊 Статистика цели` открывается без ошибки;
- `🎯 Цели -> ✏️ Редактировать цель -> 🔀 Алгоритм отправки` принимает `1/2`;
- `📄 Статусы по никам` выводится без ошибки `can't parse entities`.
