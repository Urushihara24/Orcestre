# Multi Friends Epic Bot

Telegram-бот для управления пулом аккаунтов Epic Games и рассылкой заявок в друзья по целям.

## 1. Что умеет бот

- импорт аккаунтов-отправителей (`.xlsx/.txt/.csv`);
- авторизация каждого аккаунта через Epic OAuth-ссылку и сохранение `device_auth` в БД;
- массовая смена ников Epic по файлу (`login/email -> новый ник`);
- управление несколькими целями (campaigns);
- импорт и ручное управление никами получателя внутри выбранной цели;
- планирование отправок по окнам времени с джиттером и лимитами;
- recheck (повторные проверки ранее отправленных заявок) с отдельным суточным лимитом;
- ручные операции по связям: проверка в друзьях и досыл отсутствующих;
- операции по цели: отзыв заявок, удаление из друзей, остановка текущей операции;
- статистика по цели, по никам и по отправителям по конкретному нику;
- форс-циклы отправки (с указанного аккаунта или случайного);
- переключение режимов отправки: `Новые заявки` / `Только recheck`;
- ограниченный режим доступа `auth-operator` (только список аккаунтов + авторизация по ссылке).

Бот работает через **inline-клавиатуры** (callback-кнопки).

- `📥 Импорт файлов` — импорт отправителей;

### Редактирование цели

- `🎯 На ник` — сколько отправителей нужно на 1 ник получателя в сутки;
- `⏱️ Джиттер` — `min_sec max_sec` для send-задач;
- `🕐 Окна` — активные окна цели;
- `🔁 Аккаунты на recheck` — recheck budget/сутки по отправителям (отдельно от новых send);
- `📅 Ежедневный повтор` — повторное покрытие цели на следующий день;
- `🔀 Алгоритм отправки`:
  - `1` — сначала отправитель, потом следующий (`sender_first`)
  - `2` — сначала ник, потом следующий (`target_first`)
- `🎲 Порядок отправителей`:
  - `1` — по ID аккаунтов (`ordered`)
  - `2` — случайный (`random`)
- `📋 Параметры цели` — сводка текущих параметров.

Важно: изменение параметров цели вызывает перестройку очереди send-задач (очистка старых queued/postponed + добавление актуальных).

### 4.3 Настройки

- `🛡️ API лимиты` — глобально на каждый аккаунт:
  - минимум секунд между API-запросами;
  - максимум запросов в час;
  - максимум запросов в сутки.
- `📌 Прокси` — добавить/удалить/посмотреть;
- `🧯 Новые заявки` — включить/выключить создание новых send-задач;
- `♻️ Только recheck` — оставить только recheck/досыл отсутствующих (новые send отключаются);
- `👤 Доступ auth` — список ограниченных операторов авторизации.

Примечание:
- старые глобальные кнопки `Лимит/Джиттер/Окна/TZ` отключены;
- настройка времени всегда в `Europe/Moscow`;
- актуальные параметры отправки задаются в выбранной цели.

### 4.4 Управление

- `▶️ Тик (1 раз)` — ручной одиночный цикл обработки задач;
- `📤 Экспорт` — экспорт данных;
- `⏸️ Стоп обработки` / `▶️ Старт обработки`;

## 5. Форматы ввода

### 5.1 Окна

- `24/7`
- или строки вида:
  - `days=1,2,3,4,5 from=12:00 to=20:00`
  - `days=6,7 from=10:00 to=18:00`

`days` = ISO weekday (`1..7`, где `1` — понедельник).  
Поддерживаются окна через полночь (например `22:00 -> 02:00`).

### 5.2 API лимиты

Формат:

- `min_interval_sec hourly_limit daily_limit`

Пример:

- `40 40 500`

### 5.3 Добавление аккаунта

- `login:password`

### 5.4 Массовая смена ников

- формат файла: 2 колонки
  - 1 колонка: `login/email`
  - 2 колонка: `новый ник`
- поддерживаемые форматы: `.xlsx/.txt/.csv`
- ограничения ника:
  - только `A-Za-z0-9_`
  - длина `3..16`
- если ник занят (`nickname_taken`), бот автоматически пробует похожие варианты (визуальные замены и суффиксы);
- если сработал cooldown Epic (`nickname_cooldown`) или ник невалидный, задача помечается как `failed`.

## 6. Как работает планирование

### 6.1 Send-задачи

Задачи отправки создаются по активным целям с учётом:

- выбранного алгоритма (`sender_first` / `target_first`);
- порядка отправителей (`ordered` / `random`);
- параметра `На ник`;
- суточного лимита на отправителя в цели;
- окон и джиттера;
- глобальных API-лимитов.

Дополнительно:

- перед отправкой выполняется pre-check связи (`accepted/pending`);
- если пара уже `accepted/pending`, это не считается новым покрытием;
- бот пытается подобрать следующего валидного отправителя, чтобы добрать лимит покрытия по цели.

### 6.2 Recheck-задачи

`create_recheck_tasks_job()`:

- работает только по активным целям;
- имеет **отдельный** суточный лимит на цель (`recheck_daily_limit`);
- берёт пары из уже выполненных `send_request` для ников в статусах `pending/sent/accepted`;
- не ставит дубликаты активных `check_status`;
- использует только активные аккаунты с `device_auth`.

Порядок recheck зависит от алгоритма цели:

- при `target_first` — случайный порядок ников (групп по никам);
- при `sender_first` — сохраняется последовательность отправителей, внутри отправителя ники перемешиваются.

### 6.3 Device auth защита от ошибки выбора аккаунта

При OAuth-подключении бот проверяет соответствие выбранного аккаунта и фактически авторизованного Epic-профиля:

- mismatch по email/epic_account_id не сохраняется;
- показывается понятная ошибка;
- предотвращается привязка `device_auth` к неверному аккаунту.

## 7. Статистика

### 7.1 `📊 Статистика` (главная)

Краткий срез по системе: аккаунты, цели, очередь, отправки за день, прокси.

### 7.2 `📊 Статистика цели`

Показывает:

- покрытие цели по отправителям;
- отправлено сегодня/всего;
- принято/отклонено сегодня;
- очереди send/check;
- готовые аккаунты, использованные и неиспользованные;
- упёршиеся в дневной лимит;
- блок `Запас отправителей`:
  - самый покрытый ник `X/Y`;
  - сколько осталось до исчерпания пула;
  - сколько ников уже упёрлись в пул.

### 7.3 `📄 Статусы по никам`

По каждому нику:

- статус;
- `Отпр/Прин/В друзьях/Откл`;
- очередь `send/check`;
- `Отозвано/Удалено`.

### 7.4 `👀 Отправители по нику`

Постраничный список отправителей конкретного ника с иконками статуса (`✅/❌/⏳`).

## 8. Импорт файлов

Поведение `.txt/.csv` зависит от текущего меню:

- в меню `Аккаунты` -> импорт аккаунтов;
- в меню `Массовая смена ников` -> импорт задач смены ников;
- в меню `Цели` -> импорт ников получателя в выбранную цель.

Для `.xlsx`:

- в меню `Аккаунты` -> импорт аккаунтов;
- в меню `Массовая смена ников` -> импорт задач смены ников.

## 9. ENV и обязательные переменные

Минимум для старта:

- `TELEGRAM_BOT_TOKEN`
- `ADMIN_TELEGRAM_ID` или `ADMIN_TELEGRAM_IDS`
- `DB_URL`
- Epic OAuth параметры (`EPIC_CLIENT_ID`, `EPIC_CLIENT_SECRET`, `EPIC_SWITCH_TOKEN`, `EPIC_ANDROID_TOKEN`)

Ключевые флаги:

- `DRY_RUN`:
  - `1` — безопасный тестовый режим без реальных send/check действий в worker;
  - `0` — боевой режим.
- `SEND_REQUESTS_ENABLED`:
  - `1` — разрешает реальные friend request;
  - `0` — блокирует отправку заявок.
- `APP_MODE`:
  - `all` (бот + scheduler + worker)
  - `bot`
  - `worker`
  - `scheduler`
- `LIST_PAGE_SIZE` / `TARGETS_PAGE_SIZE` / `SENDERS_PAGE_SIZE`:
  - размер страниц списков аккаунтов / ников / отправителей.

## 13. Частые проблемы

### 13.1 Postgres не стартует (`No space left on device`)

Признак: контейнер `egs-postgres` в `Restarting`, в логах ошибка записи `postmaster.pid`.

Проверка:

```bash
df -h
docker compose -f docker-compose.prod.yml ps
docker logs --tail=200 egs-postgres
```

Решение: освободить место (docker cache / system logs), затем перезапустить compose.

### 13.2 `can't parse entities`

Бот автоматически делает fallback в plain-text при проблемах Markdown. Если ошибка повторяется, проверь текст/данные, которые выводятся в сообщении.

### 13.3 Не запускаются send-задачи

Проверь:

- цель запущена (`▶️ Запустить цель`);
- обработка включена (`▶️ Старт обработки`);
- `DRY_RUN=0` и `SEND_REQUESTS_ENABLED=1`;
- у аккаунтов есть `device_auth`;
- не исчерпаны API/суточные лимиты;
- текущий момент попадает в окна цели.

### 13.4 Смена ника не проходит

Проверь:

- у аккаунта есть `device_auth`;
- новый ник соответствует формату `A-Za-z0-9_` и длине `3..16`;
- Epic не вернул `nickname_cooldown` (смена раз в ~14 дней);
- если вернулся `nickname_taken`, бот попробует похожие варианты автоматически (см. `📊 Статус смены ников`).

## 14. Важные замечания

- Кампания по умолчанию **не создаётся автоматически**. Сначала создай цель в `Менеджере целей`.
- Нумерация `№N` в UI — это порядковый номер в списке, а не `id` из БД.
- Источник истины для `device_auth` — БД; JSON-экспорт опционален (`DEVICE_AUTHS_JSON_PATH`).
